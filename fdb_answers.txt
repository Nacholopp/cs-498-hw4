Q1: FoundationDB separates transaction management from storage so each layer can scale and evolve independently. The Transaction Management System handles transaction ordering, conflict checking, and commit coordination, while the Storage System stores key-value data durably and serves reads. This separation simplifies the design and lets FDB optimize concurrency control without tying it to physical data layout.

Q2: Read-write conflicts are detected by the transaction management layer, specifically during commit processing after the client submits its mutations and read conflict ranges. The system checks whether any committed transaction wrote to data that overlaps the transaction’s read conflict ranges since its read version. If such an overlap exists, the transaction is rejected and must retry.

Q3: The read version defines the consistent snapshot of the database that a transaction reads from. All reads in the transaction are evaluated against that snapshot, which gives the transaction a stable and globally ordered view of the database. This is what allows FDB to provide strict serializable behavior when the transaction later commits successfully.

Q4: Snapshot reads do not add read conflict ranges, so they do not cause the transaction to fail if another transaction later writes the data that was read. Standard reads do participate in conflict detection and therefore protect serializability for values the transaction depends on. Snapshot reads are useful when you want to inspect data for performance or diagnostics without making the transaction conflict on those reads.

Q5: Recovery is cheap because committed mutations are first stored durably in LogServers, while StorageServers apply those mutations asynchronously in the background. Since the logs already contain the authoritative recent history, recovery does not require a heavyweight redo phase on the critical path before the system can resume. New transaction roles can be brought up and the storage layer can catch up from the logs, which keeps failover relatively fast.
